## Overall: the foundation is genuinely premium

This is not a thrown-together app. The design system (`src/theme/tokens.ts` + `src/index.css` + `tailwind.config`) is the work of someone who cares:

- **Coherent identity** — "The Steele Spreadsheet System": tiered cool-graphite surfaces (`bg → surface → elevated → overlay`), emerald "go" primary, brass accent, silver "steele", semantic + rank-medal colors. Tagline "Still Email?" ties the brand to the product's reason for existing.
- **Complete token scales** — radius (xs→2xl→full), shadows incl. `glow`/`glowAccent`, a motion system (durations, eases, spring), and a z-index scale. Single source of truth mirrored to CSS vars.
- **Self-hosted fonts with preload** — Outfit + JetBrains Mono as woff2 in `public/fonts/`, with the splash-critical JetBrains weight `<link rel=preload>`'d in `index.html` for LCP. No render-blocking Google CDN.
- **Light + dark themes** both defined in `index.css` (`:root` dark, light variant from line 141).
- **Accessibility + motion maturity** — `@media (prefers-reduced-motion: reduce)` in CSS *and* `useReducedMotion()` honored in components (`PageTransition`, `save-button`, `score-input`, several pages). `jest-axe` accessibility tests exist (`test/accessibility.test.tsx`).
- **State coverage** — dedicated `EmptyState`, `Skeleton`, `ErrorBoundary`, `OfflineBanner` components; pages use them (e.g. leaderboard/invites empty states, skeleton loaders).
- **Motion moments** — `framer-motion` page transitions, animated score inputs and save buttons.

The bones are excellent. The findings below are about *consistency and finish*, not foundation.

## Findings

### U-FIX-1 — PIN entry is inconsistent AND silently capped at 4 digits
**Severity: HIGH (user-flagged + a real functional bug behind it).**
**This is the user's "confirm PIN is text, not PIN — make it consistent" item. The root cause is deeper than cosmetic.**
**VISUALLY CONFIRMED 2026-05-30** (local dev `/signup` screenshot): "CHOOSE A PIN (4–8 DIGITS)" renders as the polished 4-cell segmented `PinInput`; "CONFIRM PIN" directly below renders as a plain boxed password input (`••••`). On screen they read as two different design languages stacked, and the segmented field shows exactly 4 cells — contradicting the "4–8 digits" label. Login uses the same 4-cell `PinInput`.

On **SignupPage** (`pages/SignupPage.tsx`):
- Label: **"Choose a PIN (4–8 digits)"** (line 151).
- First PIN: `<PinInput>` (line 152) — the premium 4-cell segmented component.
- Confirm PIN: a plain `<Input type="password" maxLength={8}>` (lines 157–169).

Two problems:
1. **Visual inconsistency (what the user noticed):** the first field is the polished segmented PIN; the second is an ordinary boxed input. They look like two different products stacked on top of each other.
2. **Functional bug (the deeper issue):** `PinInput` is **hard-coded to exactly 4 cells** (`components/PinInput.tsx`: `length: 4`, `i < 3`, paste `slice(0, 4)`). So despite the "4–8 digits" copy and the confirm field accepting up to 8, the entry field **physically cannot accept more than 4 digits**. A user who types/pastes more in confirm can never match, and PINs are silently forced to 4.

Also: `PinInput` sets `autoComplete="current-password"` on cell 0 (`PinInput.tsx:68`) — wrong for *signup* (should be `new-password`); it's a login-originated component reused without adjusting intent.

On **JoinPage** (`pages/JoinPage.tsx`): **both** PIN and confirm are plain `<Input type="password">` (lines 219–245). Internally consistent, but it doesn't use the premium `PinInput` at all — so the invite-accept flow (the path most new users actually take) looks *less* polished than signup.

**Recommendation — DECIDED 2026-05-30: PIN range is 4–8 digits (variable):**
- Make `PinInput` variable-length (accept a `length`/`max` prop, render N cells, allow paste up to max) and use it for **both** entry and confirm on Signup, Join, **and** Login.
- Fix `autoComplete` to `new-password` in signup/join contexts.
- Net goal: one variable-length PIN component, used identically everywhere a PIN is entered or confirmed.

### U-FIX-2 — Remove the now-pointless email capture in the invite flow
**Severity: MEDIUM (user-flagged). CONFIRMED 2026-05-30: this is the "Invitee email" shown when generating an invite link.**

The **"Invitee email (optional)"** field in the Create-Invite card on `pages/LeagueAdminInvitesPage.tsx` (lines 83–93), helper text "Leave blank for a generic shareable link." The invite flow already generates and auto-copies a join link (`/join/{token}`), so the email field is dead weight — corroborated by the repo's E2 note (*"invite link flow makes email unnecessary"*) and the "Still Email?" tagline.

**Fix:** drop the `inviteeEmail` state + field (lines 17, 31, 37, 83–93), post an empty invite body, and remove the "For: {invitee_email}" line from the invite list (lines 122–126). Check whether the backend `invitee_email` column/param should also be retired or left nullable for back-compat.

### U1 — JoinPage polish lags Signup/Login
**Severity: MEDIUM.**
The invite-accept page is where most new players land. It uses plain inputs (U-FIX-1), and should get the same `PinInput`, spacing, and motion treatment as Signup so a new user's *first* impression matches the premium bar. Folds into the U-FIX-1 fix.

### U2 — Stale display-font token contradicts the real render config
**Severity: LOW (cleanup / clarity; not a visible regression).**
`theme/tokens.ts:75` declares `display: '"Instrument Serif", Georgia, serif'`, but no Instrument Serif font is loaded (`public/fonts/` has only Outfit + JetBrains Mono) and the Tailwind `font-display` utility actually **aliases to Outfit** (`tailwind.config`, with a comment saying so). So nothing renders in Instrument Serif or its Georgia fallback — but the token file, billed as "single source of truth," disagrees with the config. Remove or correct the token so future contributors aren't misled.

### U3 — `window.confirm()` for destructive actions breaks the premium feel
**Severity: LOW–MEDIUM.**
Several destructive actions use the native browser `confirm()` dialog — `LeagueMembersPage` (remove member / leave league, lines 56/70), `admin/PlayersPage` (line 62), `admin/InvitesPage` (line 98). Native dialogs are jarring against an otherwise bespoke UI. There's already a styled `ui/dialog.tsx` and a "type the name to confirm" pattern in `LeagueSettingsPage`. Recommend routing destructive confirms through the design-system dialog for consistency.

### U4 — Low-res "partnership" logo undercuts the premium splash
**Severity: LOW–MEDIUM (premium-feel; needs a product/intent decision). FOUND 2026-05-30 in the live visual pass.**
The Login splash shows **"IN PARTNERSHIP WITH" above a Robinsons (squash) logo**, with the "Still Email?" tagline beneath. The wordmark above it ("THE STEELE / SPREADSHEET SYSTEM") is crisp, wide-tracked vector type; the Robinsons logo immediately under it is a **low-resolution raster with visibly pixelated edges** — the single biggest quality drop on the otherwise-premium auth screen. **DECIDED 2026-05-30: the joke stays — improve quality (batch U8).** Actions:
- Replace the low-res bitmap with a crisp/transparent asset (ideally vector or 2–3× raster) so it stops reading as "pasted clipart" against the vector wordmark; align its sizing/spacing to the splash grid.
- Unify the splash lockup: the **Signup** splash shows only the wordmark (no partnership line/tagline) while Login shows the full lockup — apply the same lockup on both.
- Informational only (not blocking the private soak): embedding a real third-party brand mark carries a small trademark/professionalism consideration if the app ever goes more public.

## Coverage — what this pass did and did NOT cover

**Audited (code-level):** design tokens, global CSS, font loading, `PinInput`, `Brand`, the auth flow (Login/Signup/Join), the invite flow, destructive-action patterns, motion/a11y posture.

**Visually assessed 2026-05-30 (live, local dev):** the **Login** and **Signup** pages at desktop width. Verdict: premium foundation holds on screen — wide-tracked gold wordmark, clean tiered-graphite card, emerald CTA, segmented PIN. Findings surfaced live: U-FIX-1 (confirmed) and U4 (low-res partnership logo).

**Visually verified 2026-05-30 (live, full local stack, authenticated as a seeded player with real predictions + scored matches):** Dashboard, Predictions, Leagues hub, and per-league leaderboard. Verdict — the data pages **deliver on the premium bar**: filled Predictions grid (large score inputs, flag emojis, FT + points badges), correct scoring render (exact score → **10 pts**), correct **timezone conversion** (19:00 UTC kickoff → "THU 11 JUN, 20:00" in Europe/London), graceful empty states, and a clean per-league standings table ("Your position #1 · 23 pts"). The "blown away" bar is met by the pages seen.

**Config footgun found (not a product bug):** the empty-data states first observed were because the dev `apps/web/.env.local` sets `VITE_API_URL` to the **production** Railway API — so a local `pnpm dev` frontend talks to (empty) prod unless repointed at the local API. Worth a comment in the file or a `.env.development` default; low priority, but it will confuse anyone doing local full-stack dev. (Reverted after testing.)

**Still pending:** Bracket/Knockout + Specials pages (need knockout matches / specials data seeded), Compare, and an explicit 375px-mobile + light-theme pass.
